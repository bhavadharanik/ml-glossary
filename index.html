1. ML Fundamentals

How Machine Learning Works: Training Data ──> [ Model ] ──> Predictions │ ▲ │ │ │ │ └── features ──> Learning Algorithm <── loss (error) (gradient descent) Analogy: Like TDD but automated Input examples = test cases Model = your function Training = auto-fixing the function until tests pass Loss = number of failing tests
Machine Learning
A program that learns patterns from data instead of being explicitly programmed with rules.
>> Instead of writing if/else rules, you show the system 10,000 examples and it figures out the rules itself.
CRITICAL
Model
A mathematical function that takes input and produces a prediction. Has parameters (weights) that are learned during training.
>> Like an API endpoint with learned logic instead of hand-written logic. Input goes in, prediction comes out.
Training
Showing a model many examples so it adjusts its parameters to make better predictions. Minimizes a loss function.
>> Like running your function against thousands of test cases, auto-tweaking parameters each time until outputs match expected results.
Inference
Using a trained model to make predictions on new, unseen data. This is what happens in production.
>> Like calling your deployed API in production. Training = development, inference = production.
Loss Function
Measures how wrong the model's predictions are. MSE for regression, cross-entropy for classification. Goal: minimize this.
>> Like an error metric in your Grafana dashboard. Lower = better. Training tries to push this number down.
CRITICAL
Gradient Descent
Algorithm that adjusts model parameters to reduce loss. Calculates which direction to change each parameter and by how much.
>> Your model is wrong. The loss tells you HOW wrong. The gradient tells you WHICH DIRECTION to fix. The learning rate tells you HOW MUCH to fix. Like debugging with a compass.
CRITICAL
Overfitting
Model memorizes training data but fails on new data. High training accuracy, low test accuracy.
>> Like hardcoding "if requestID == 42 return success". Works during testing, useless in production.
CRITICAL
Underfitting
Model is too simple to capture patterns. Low accuracy on both training and test data.
>> Like writing a single if/else rule when you need a complex decision tree. Too simple for the problem.
Supervised Learning
Training with labeled data: input AND correct answer. Classification and regression are supervised.
>> Like unit tests with expected outputs. The model learns to match input to correct output.
CRITICAL
Unsupervised Learning
Training without labels: model finds patterns/groups on its own. Clustering is unsupervised.
>> Like grouping log entries by pattern without pre-defining categories — the system finds natural clusters.
Train / Test / Validation Split
Training (70-80%): model learns. Validation (10-15%): tune hyperparameters. Test (10-15%): final evaluation. NEVER touch test during training.
>> Like dev/staging/production environments. Build on dev, validate on staging, final check is production — never skip stages.
Feature
An individual measurable property used as input to the model. Age, income, message_count are all features.
>> Like parameters in an API request. Each feature is one field the model uses to decide.
Feature Engineering
Creating new features from raw data. E.g., extracting "day_of_week" from a timestamp, or "messages_per_hour" from a count and duration.
>> Like adding computed columns to a database view: price_per_sqft = price / sqft. Transforms raw data into more informative signals.
INTERVIEW
Hyperparameters
Settings YOU choose before training (learning rate, batch size, number of layers). Not learned by the model itself.
>> Like config values in your Docker environment variables or Helm values.yaml. You set them; the system uses them.

2. Evaluation Metrics

Confusion Matrix: Predicted: YES Predicted: NO Actual: YES │ True Positive │ False Negative │ Actual: NO │ False Positive │ True Negative │ Precision = TP / (TP + FP) "How trustworthy are my alerts?" Recall = TP / (TP + FN) "How many real problems did I catch?" F1 = 2*P*R / (P+R) "Balance of both" Example: fraud detection with 99% non-fraud data Model that always says "not fraud" = 99% accuracy, F1 ≈ 0 (useless) Good model: 85% accuracy, F1 = 0.78 (actually works)
Accuracy
(TP + TN) / Total. Percentage of correct predictions. MISLEADING with imbalanced data.
>> A fraud system that always says "not fraud" is 99% accurate but catches ZERO fraud. Always check F1 too.
CRITICALINTERVIEW
Precision
TP / (TP + FP). Of everything predicted positive, how many were actually positive? Optimize when false positives are costly.
>> Your monitoring system raises 10 alerts. 8 were real incidents. Precision = 80%. "How trustworthy are my alerts?"
CRITICAL
Recall (Sensitivity)
TP / (TP + FN). Of all actual positives, how many did we catch? Optimize when missing positives is costly (cancer detection, fraud).
>> 20 real security incidents happened. Your SIEM caught 8. Recall = 40%. "How many real problems did I detect?"
CRITICAL
F1 Score
2 * (P * R) / (P + R). Harmonic mean of precision and recall. Use when you need balance between the two, especially with imbalanced classes.
>> On imbalanced data, high accuracy can mask a broken model. F1 exposes it: a classifier that ignores the minority class will show F1 near 0.
CRITICALINTERVIEW
AUC-ROC
Area Under the ROC Curve. Plots True Positive Rate vs False Positive Rate at all thresholds. 1.0 = perfect, 0.5 = random.
>> Like a performance graph showing how well your system separates signal from noise across all sensitivity settings — threshold-independent.
Class Imbalance
When one class dominates (99% not-fraud, 1% fraud). Models predict the majority class and look accurate but are useless.
>> A model trained on server logs where 99% are INFO and 1% are ERROR will often just predict INFO for everything. Fix with SMOTE oversampling, class weights, or focal loss.
CRITICALINTERVIEW
Precision@K / Recall@K
Retrieval metrics evaluated at the top K results. Precision@K = relevant items in top K / K. Recall@K = relevant items in top K / all relevant items.
>> Like asking: of the top 10 search results returned, how many were actually useful? P@10 tells you your top-10 quality.
INTERVIEWPRODUCTION
NDCG (Normalized Discounted Cumulative Gain)
Ranking metric that rewards relevant results appearing higher in the list. Discounts relevance by position. 1.0 = perfect ranking.
>> Search engines care not just whether relevant results appear but whether they appear FIRST. NDCG penalizes a correct result buried at position 8 vs position 1.
INTERVIEW
MRR (Mean Reciprocal Rank)
Average of 1/rank of the first relevant result across queries. If the first correct answer is at position 3, score = 1/3.
>> In a Q&A system, MRR measures whether the right answer tends to appear near the top. Position 1 scores 1.0, position 2 scores 0.5, position 5 scores 0.2.
INTERVIEW

3. Classical ML Models

When to use which model: Tabular data? ├── Need interpretability? ──> Decision Tree / Logistic Regression ├── Best accuracy? ──> XGBoost (industry standard) ├── Good default? ──> Random Forest └── Predict a number? ──> Linear Regression No labels? └── Group similar items? ──> K-Means Clustering Text/Images/Sequence? └── Neural Networks / Transformers
Linear Regression
Predicts continuous values: y = w1*x1 + w2*x2 + ... + b. Minimizes MSE. Assumes a linear relationship between inputs and output.
>> y = mx + b but with many features. Predict house price from size, bedrooms, and location.
Logistic Regression
Classification (despite the name). Applies sigmoid to a linear combination of features to output a probability 0-1. Threshold at 0.5 by default.
>> Takes any linear score, squashes it to 0-1 via sigmoid. Output above 0.5 = positive class. Simple and interpretable — often required in regulated industries.
UK MARKET
Decision Tree
If/else splits on features: "age > 30?" then "income > 50K?". Uses Gini impurity to choose splits. Human-readable but overfits easily.
>> Like a flowchart or nested switch statement. Easy to explain to a non-technical stakeholder, but memorizes data if too deep.
Random Forest
100-1000 decision trees on random data subsets. Final prediction = majority vote. Ensemble method. Resistant to overfitting.
>> Like 100 microservice replicas each seeing slightly different traffic, then taking a consensus vote. More reliable than any single replica.
CRITICAL
XGBoost / Gradient Boosting
Trees built SEQUENTIALLY: each tree fixes errors of the previous one. Tree 2 predicts the residual errors of Tree 1. Industry standard for tabular data.
>> Like iterative hotfix deployments: each release patches exactly what the previous one got wrong. High accuracy, needs careful tuning.
CRITICALINTERVIEW

Random Forest (Bagging)

Trees trained INDEPENDENTLY on random subsets. Vote at the end. Good default, hard to mess up. Lower variance.

XGBoost (Boosting)

Trees trained SEQUENTIALLY, each fixing previous errors. Higher accuracy but needs more tuning. Watch for overfitting.

4. Deep Learning & Neural Networks

Neural Network: Input Layer Hidden Layer 1 Hidden Layer 2 Output [feature 1] ──> [ neuron ] ──> [ neuron ] ──> [prediction] [feature 2] ──> [ neuron ] ──> [ neuron ] ──> [feature 3] ──> [ neuron ] ──> [ neuron ] Each connection has a weight Each neuron: activation( weights * input + bias ) Training loop: 1. Forward pass ──> get prediction 2. Calculate loss ──> how wrong? 3. Backward pass ──> compute gradients (backpropagation) 4. Update weights ──> weights -= learning_rate * gradient 5. Repeat 1000x
Neural Network
Layers of neurons. Input -> hidden layers -> output. Each neuron: weighted sum + activation function. Deep networks have many hidden layers.
>> Like a multi-layer microservice pipeline. Data flows through each service layer, getting transformed and enriched at each step.
Backpropagation
After forward pass, calculate loss, then flow error BACKWARDS using chain rule to compute each weight's contribution. Then update weights proportionally.
>> Like distributed tracing in reverse: an error is traced back through each service layer to find which component caused it, then each is patched proportionally.
CRITICAL
ReLU (Rectified Linear Unit)
max(0, x). If positive, pass through unchanged. If negative, output 0. Most common activation function for hidden layers.
ReLU(5) = 5, ReLU(-3) = 0, ReLU(0.5) = 0.5
Sigmoid
1 / (1 + e^(-x)). Squashes any real number to the range 0-1. Used for binary classification output layers.
>> Any score, no matter how large or negative, gets compressed into a 0-100 percentage. Natural output for "probability of class 1".
CRITICAL
Dropout
During training, randomly set a fraction (typically 20-50%) of neurons to 0. Forces the network to be resilient. Turned OFF during inference.
>> Like chaos engineering in production: randomly killing pods during load testing forces the system to build in redundancy. Disabled on the final deployment.
Transfer Learning
Starting from a model pretrained on a large dataset (ImageNet, large text corpus) and fine-tuning on your specific task. Dramatically reduces training time and data needs.
>> Like forking a mature open-source library instead of writing from scratch. The pretrained model already knows low-level features; you just adapt the top layers.
INTERVIEWPRODUCTION
Convolutional Neural Network (CNN)
Specialized architecture for images. Convolutional layers detect local patterns (edges, shapes) using sliding filters. Pooling reduces spatial dimensions.
>> Like sliding a pattern-matching function across a 2D grid — each filter is a small detector that fires when it sees its pattern anywhere in the image.
INTERVIEW
Batch Normalization
Normalizes activations within each mini-batch during training. Stabilizes and speeds up training. Reduces sensitivity to weight initialization.
>> Like standardizing API response formats across microservices — ensures each layer receives consistently-scaled inputs, preventing training instability.

5. NLP (Natural Language Processing)

Tokenization
Breaking text into tokens. "unhappiness" -> ["un", "happi", "ness"]. BPE (Byte-Pair Encoding) most common in modern LLMs. Models see tokens, not words.
>> Like splitting a URL into path segments. The model processes integer token IDs; raw text is the human-readable abstraction.
Embedding
Dense vector representation. Similar items map to similar vectors. "king" - "man" + "woman" ≈ "queen". Typically 512-1536 dimensions for modern models.
>> Like a content-aware hash: similar inputs produce similar outputs (unlike a cryptographic hash). A 768-dim fingerprint that encodes semantic meaning.
CRITICAL
Cosine Similarity
Measures the angle between two vectors. cos(A,B) = (A·B) / (|A|*|B|). 1.0 = identical direction, 0 = orthogonal (unrelated). Core of RAG retrieval.
cos([1,0], [0,1]) = 0.0 (perpendicular, unrelated) cos([1,0], [1,0]) = 1.0 (identical) cos([1,1], [1,0]) = 0.71 (somewhat similar)
CRITICAL
TF-IDF (Term Frequency-Inverse Document Frequency)
Classic term-weighting for information retrieval. TF: how often a word appears in a document. IDF: penalizes common words ("the", "is"). High TF-IDF = rare word that appears often in this doc.
>> Like finding a microservice's unique signature: a word that appears 10x in one doc but rarely elsewhere is a strong discriminator — common words like "the" are noise.
INTERVIEW
BM25
Probabilistic extension of TF-IDF. Industry standard for keyword/lexical search. Accounts for document length. Used in Elasticsearch by default.
>> The workhorse behind most production search systems before neural embeddings. Fast, no GPU, interpretable. Still competitive for exact keyword queries.
INTERVIEWPRODUCTION
Named Entity Recognition (NER)
Identifying and classifying named entities in text: people, organizations, locations, dates, monetary values.
>> Like a regex parser but learned from data. "Apple reported £50B revenue in London" → Apple=ORG, £50B=MONEY, London=LOC.

6. Transformers & Attention

Self-Attention: How Transformers Understand Context Sentence: "The cat sat on the mat because it was tired" For the word "it", attention looks at ALL other words: "The" -> low attention (0.02) "cat" -> HIGH attention (0.65) <-- "it" refers to "cat"! "sat" -> low attention (0.08) "mat" -> low attention (0.05) "tired" -> medium attention (0.20) This is how the model knows "it" = "cat" The Math: Q (Query) = "What am I looking for?" <-- like a WHERE clause K (Key) = "What do I contain?" <-- like a database index V (Value) = "What info do I provide?" <-- like the returned data Attention = softmax(Q * K^T / sqrt(d)) * V
Transformer
Architecture behind all modern LLMs. Processes all tokens simultaneously using self-attention. Replaced sequential RNNs/LSTMs. Introduced in "Attention Is All You Need" (2017).
>> GPT, Claude, Gemini, Llama — all Transformers under the hood. THE architecture of modern AI. Understanding it is non-negotiable for ML interviews.
CRITICAL
Self-Attention
Each token attends to all other tokens to build context-aware representations. Uses Query, Key, Value projections. Runs in parallel across all positions.
>> Like a service that queries every other service in the cluster before building its response, weighting the most relevant ones more heavily.
CRITICAL
Multi-Head Attention
Run attention multiple times in parallel with different learned projections (heads). Each head can capture different relationship types. Outputs are concatenated.
>> Like running multiple independent feature extractors in parallel — one head might focus on syntax, another on coreference, another on semantics.
INTERVIEW
Positional Encoding
Transformers have no inherent sense of order. Positional encodings inject position information into token embeddings so the model knows word order.
>> Like adding a sequence number to messages in a distributed queue. Without it, the model would treat "dog bites man" and "man bites dog" identically.

Encoder (BERT)

Reads text bidirectionally. Creates rich contextual representations. Best for: classification, NER, semantic search, understanding tasks.

Decoder (GPT/LLaMA)

Generates text autoregressively, one token at a time (left to right). Best for: text generation, chat, completion, summarization.

7. LLMs & Generative AI

How an LLM Generates Text: Input: "The cat sat on the" Step 1: Model sees all tokens, computes probabilities: "mat" 0.35 <-- highest "floor" 0.20 "bed" 0.15 "chair" 0.10 ... ... Step 2: Sample next token based on temperature: temp=0.0 -> always pick "mat" (deterministic) temp=0.7 -> usually "mat", sometimes "floor" temp=1.5 -> could pick anything (creative/random) Step 3: Append "mat", repeat from Step 1: "The cat sat on the mat" -> predict next -> "." (0.45) This is autoregressive generation: one token at a time
Large Language Model (LLM)
Very large transformer (billions of parameters) pretrained on massive text data to predict the next token. Forms the basis of modern AI assistants.
>> GPT-4, Claude, Gemini, LLaMA are all LLMs. They just predict the next word — extremely fast, at massive scale, with emergent reasoning capabilities.
CRITICAL
Temperature
Controls output randomness. 0 = deterministic (always highest probability token). 1+ = creative/random. Technically divides logits before softmax.
>> Low temperature for factual tasks (Q&A, code generation). Higher temperature for brainstorming, creative writing. Production systems for finance or compliance: use temp 0.
CRITICALINTERVIEW
Prompt Engineering
Designing prompts for better output. Zero-shot (no examples), few-shot (2-5 examples), chain-of-thought (step-by-step reasoning), system prompts.
>> Like writing a well-structured API contract: the more precisely you specify input format and expected output, the more reliably the LLM performs.
CRITICALINTERVIEW
Hallucination
LLM generates confident but factually incorrect content. A fundamental challenge with autoregressive models. Mitigated by RAG, grounding, and output validation.
>> Like a microservice that returns a 200 OK with plausible-looking but wrong data. No exception thrown — you need explicit validation to catch it.
CRITICALUK MARKET
Context Window
Maximum number of tokens the model can "see" at once. GPT-4: 128K, Claude 3: 200K. Beyond this, information is not available to the model.
>> Like RAM for the model's working memory. Larger context = higher cost and latency. Production systems must chunk and summarize long documents strategically.
PRODUCTION
Structured Output / JSON Mode
Constraining LLM output to valid JSON or a defined schema. Avoids brittle regex parsing. Modern APIs (OpenAI, Anthropic) support this natively.
>> Like enforcing a response DTO instead of free-text. Your downstream service can safely deserialize the response without try/except around JSON parsing.
PRODUCTIONINTERVIEW
Guardrails for LLMs
Input/output validation layers around LLM calls. Check for harmful content, off-topic responses, PII leakage, schema violations. Can be rule-based or classifier-based.
>> Like middleware in an Express or FastAPI app: every request passes through validation before reaching the handler, and every response is checked before returning to the client.
PRODUCTIONUK MARKET

Fine-tuning

Change model behavior. Needs labeled training data + compute. Risk: catastrophic forgetting. Best for: specific style, format, domain terminology.

RAG

Add external knowledge at inference. No training needed, up-to-date info. Bottleneck: retrieval quality. Best for: company docs, knowledge bases, dynamic data.

8. RAG & Agentic AI

RAG Pipeline (Retrieval Augmented Generation): OFFLINE (once): Documents ──> Chunk (500 tokens) ──> Embed (sentence-transformers) ──> Store in Vector DB (FAISS / Pinecone) ONLINE (per query): User Question ──> Embed ──> Search Vector DB (cosine similarity) ──> Top 3-5 chunks retrieved ──> Build prompt: [context + question] ──> LLM generates grounded answer Hybrid: BM25 keyword search + dense vector search + RRF reranking
RAG (Retrieval Augmented Generation)
Embed docs -> store in vector DB -> embed query -> find similar chunks -> pass to LLM -> grounded answer. Reduces hallucination without fine-tuning.
>> Like searching Stack Overflow or internal docs before answering a question, instead of guessing from memory. The LLM becomes a smart formatter of retrieved facts.
CRITICALINTERVIEW
Vector Database — FAISS vs Pinecone vs ChromaDB vs Weaviate
Databases optimized for approximate nearest-neighbor search on high-dimensional embeddings. Each has different trade-offs for production use.
>> Like choosing a database engine: SQLite vs PostgreSQL vs managed RDS. Pick based on scale, operational overhead, and query needs.
FAISS — local, no server, fastest for prototype/batch; no persistence by default Pinecone — fully managed, serverless, easy production; costs money ChromaDB — lightweight managed, good for small-medium projects, open source Weaviate — GraphQL API, built-in hybrid search, good for production self-host
INTERVIEWPRODUCTION
Hybrid Search / Reciprocal Rank Fusion (RRF)
Combining dense vector search (semantic similarity) with sparse keyword search (BM25). RRF merges ranked result lists without needing score normalization.
>> Like running two different search algorithms and combining their rankings. RRF gives each result a score of 1/(rank + 60), summed across both lists — simple and effective.
INTERVIEWPRODUCTION
Agent
LLM that can DECIDE what to do and USE TOOLS to act. Not just text generation — it reasons, selects tools, executes them, and iterates based on results.
>> Like an automated on-call engineer: it reads the alert, decides which runbook to follow, calls the relevant APIs, checks the output, and either resolves or escalates.
CRITICALINTERVIEW
Function Calling / Tool Use
Define a set of tools (functions with JSON schemas) the LLM can invoke. The model returns a tool call with arguments; your code executes it and returns results to the model.
>> Like defining an OpenAPI spec for the LLM. The model picks the right endpoint and constructs the correct request body. You handle execution and return the HTTP response.
INTERVIEWPRODUCTION
LangGraph
Graph-based framework for building stateful LLM agent workflows. Nodes = steps, edges = conditional transitions, shared state = TypedDict. More controllable than chains.
>> Like defining a state machine for your agent: each node is a handler, edges are conditional routing rules, and the state object is passed between nodes like a context object.
INTERVIEW
Chunking Strategy
How you split documents before embedding. Fixed-size (500 tokens), sentence-aware, paragraph-based, or semantic chunking. Chunk overlap (e.g., 50 tokens) prevents context loss at boundaries.
>> Like pagination in an API: if your page breaks mid-sentence you lose context. Overlap is like returning the last item of page N at the start of page N+1 for continuity.
PRODUCTIONINTERVIEW

9. MLOps & Production

ML Pipeline in Production: Data ──> Preprocess ──> Feature Eng ──> Train ──> Evaluate ──> Deploy ──> Monitor │ │ S3 / Data Warehouse Prometheus Feature Store Grafana Data Validation Alerts / Retraining
Batch vs Real-time Inference
Batch: process large data on a schedule (nightly or weekly). Real-time: per-request serving (<100ms SLA, like fraud detection or recommendation APIs).
>> Batch = cron job or Airflow DAG that processes yesterday's data. Real-time = a REST endpoint that must respond before the page loads. Different cost profiles, different infra.
INTERVIEWPRODUCTION
Data Drift
Input data distribution shifts over time. A model trained on last year's data may fail silently on today's inputs.
>> Like an API contract change in an upstream service that nobody told you about. The schema looks the same but the values mean something different now.
CRITICALPRODUCTION
Model Drift / Concept Drift
Model performance degrades over time because the underlying relationship between features and labels changes. Requires monitoring and scheduled retraining.
>> Like a Grafana alert for rising error rate: you need to monitor prediction metrics in production, not just at training time. Set up automated retraining triggers.
PRODUCTION
Feature Store
Centralized repository for computed features. Ensures training and serving use identical feature pipelines. Prevents training-serving skew. (Feast, Tecton, Vertex AI Feature Store)
>> Like a shared Redis cache for ML features: instead of every team recomputing "user_30day_spend", it's computed once and served consistently to both training and inference.
PRODUCTION
Quantization
Reduce model weight precision: FP32 -> INT8 (4x smaller, faster). Post-training quantization (PTQ) or quantization-aware training (QAT). Slight quality tradeoff.
>> Like compressing Docker images with multi-stage builds: smaller artifacts, faster pulls and deployments, minimal functional difference in most cases.
PRODUCTION
Kubernetes for ML / Model Serving
Running models in production at scale. vLLM for high-throughput LLM serving (continuous batching, paged attention). TorchServe for PyTorch model APIs. KServe for Kubernetes-native serving.
>> vLLM is to LLM serving what NGINX is to web serving: purpose-built for throughput. Continuous batching means multiple requests share a forward pass — dramatically improves GPU utilization.
PRODUCTIONINTERVIEW
A/B Testing & Shadow Mode
A/B: route traffic between model versions to compare metrics in production. Shadow mode: new model runs in parallel but doesn't affect users — compare outputs before rollout.
>> Like canary deployments in Kubernetes: route 5% of traffic to the new model, monitor business metrics, roll forward if no regression.
PRODUCTIONINTERVIEW
Model Registry & MLflow
Centralized store for trained models, versions, metrics, and parameters. MLflow tracks experiments, logs artifacts, manages model lifecycle (staging/production/archived).
>> Like a container registry (Docker Hub / ECR) but for ML models. Every trained model is a versioned artifact with metadata, reproducible from logged parameters.
PRODUCTION

10. Ethics, Fairness & UK Market

GDPR & Data Privacy in ML
UK GDPR / EU GDPR compliance requirements for ML systems: lawful basis for processing, data minimization, right to erasure (right to be forgotten), purpose limitation.
>> GDPR impacts your training data pipeline: you cannot train on personal data without a lawful basis. "Right to erasure" means you may need to retrain or implement machine unlearning.
UK MARKETCRITICAL
Responsible AI / AI Ethics
Framework for building AI that is safe, fair, transparent, and accountable. Pillars: fairness, reliability, privacy, transparency, inclusiveness, accountability. UK government's AI Safety Institute focuses on this.
>> Like software engineering's secure-by-design principle, but applied to model behavior and societal impact. UK employers increasingly ask about this — especially in finance, health, and public sector.
UK MARKETINTERVIEW
Explainability vs Interpretability
Interpretability: model structure is inherently understandable (decision trees, linear models). Explainability: black-box model explained post-hoc (SHAP, LIME, attention maps, GradCAM).
>> Interpretable = open-source code you can read. Explainable = black-box service with good logging and observability. UK finance (FCA) requires explainable credit decisions — "why was this loan rejected?"
UK MARKETINTERVIEW
SHAP (SHapley Additive exPlanations)
Assigns each feature a contribution value for a specific prediction. Based on game-theoretic Shapley values. Consistent and locally accurate.
>> Like a P&L attribution report: instead of just the bottom line, you see how much each input feature contributed positively or negatively to this specific prediction.
INTERVIEWUK MARKET
Model Cards
Structured documentation for ML models: intended use, out-of-scope uses, evaluation results across demographic groups, training data description, limitations. Google introduced; increasingly standard.
>> Like a technical data sheet / README for a model artifact. UK public sector and large enterprises increasingly require these before deploying ML systems.
UK MARKETPRODUCTION
Bias & Fairness in ML
Fairness metrics: demographic parity (equal positive rate across groups), equalized odds (equal TPR/FPR), calibration. Bias sources: historical data, label quality, feature proxies.
>> A hiring model trained on historical promotions learns to replicate past gender bias. Auditing for fairness means checking whether error rates differ systematically across protected groups.
UK MARKETINTERVIEW
UK AI Safety Institute / AI Act Awareness
UK established the world's first AI Safety Institute (2023). EU AI Act classifies AI by risk level: unacceptable (banned), high-risk (strict requirements), limited-risk, minimal risk.
>> High-risk AI in UK/EU (healthcare, credit scoring, hiring) requires conformity assessments, logging, human oversight — similar to how medical devices need CE marking and rigorous documentation.
UK MARKETINTERVIEW

11. Math Essentials

Vector
A list of numbers: [3, 4, 5]. Embeddings are high-dimensional vectors (768-1536 dims). Core data structure of all ML.
[3, 4] is a 2D vector. A 768-dim embedding is just a list of 768 floats in memory.
Dot Product
Multiply corresponding elements and sum: [1,2,3] · [4,5,6] = 4+10+18 = 32. Core operation in attention mechanisms and similarity computation.
np.dot([1,2,3], [4,5,6]) = 32
Standard Deviation & Z-Score
Standard deviation measures spread from the mean. Z-score: z = (value - mean) / std. Tells you how many standard deviations a value is from the norm.
>> Like a p-value for outlier detection: a z-score of 3 means this value is 3 standard deviations above normal — likely an anomaly worth flagging.
Softmax
Converts a vector of arbitrary numbers into a probability distribution that sums to 1.0. Used in multi-class classification and attention.
softmax([2.0, 1.0, 0.1]) = [0.71, 0.26, 0.09] (sums to 1.0)
CRITICAL
Matrix Multiplication
Core operation in neural networks. Every forward pass is largely matrix multiplications. Shape rules: (m, n) × (n, p) = (m, p). GPUs are optimized for this.
>> Like bulk SQL joins: instead of one query at a time, you batch thousands of dot products in a single GPU kernel. Understanding shapes prevents runtime errors.

12. Interview Preparation

STAR Format for ML Projects

Use this structure for any project question

Situation: Context — what was the business problem or technical challenge? What system or product was involved?

Task: What were you specifically responsible for? What did you own end-to-end?

Action: What technical decisions did you make and why? What alternatives did you consider? What trade-offs did you accept?

Result: Quantified outcome — latency improvement, accuracy gain, cost reduction, business impact. If no metric, explain why and what proxy you used.

Tip: End every project story with what you would improve given more time. It shows awareness and continued thinking.

Common ML System Design Questions

Frequently asked in UK ML Engineer interviews

1. Design a recommendation system — Cover: data pipeline, feature engineering, candidate generation (ANN/embedding), ranking model, A/B testing, monitoring for CTR/diversity.

2. Design a fraud detection system — Cover: real-time vs batch, feature store, imbalanced classes, threshold tuning, explainability for compliance, feedback loops.

3. Design a RAG-based Q&A system — Cover: chunking, embedding model choice, vector DB selection, hybrid search, reranking, context window limits, hallucination guardrails.

4. How do you monitor a model in production? — Cover: data drift detection, prediction distribution monitoring, business KPIs, alerting thresholds, retraining triggers.

Common Interviewer Follow-up Questions

Questions interviewers ask after you explain a project

"How did you handle class imbalance?" — SMOTE, class weights, focal loss, threshold adjustment, choosing F1 over accuracy.

"How would you scale this?" — Feature store, batch vs streaming inference, quantization, async processing, caching embeddings.

14. Self-Quiz (Click to Reveal)

Test Yourself

What is the difference between precision and recall? When do you optimize for each?
Precision = of my positive predictions, how many were correct? Recall = of all actual positives, how many did I catch? Optimize precision when false positives are costly (spam filters — wrongly blocking a legitimate email is bad). Optimize recall when false negatives are costly (cancer detection — missing a real case is worse).
A model has 99% accuracy on fraud detection. Is it good? How would you evaluate it properly?
Not necessarily. If 99% of transactions are legitimate, a model that always predicts "not fraud" achieves 99% accuracy but catches zero fraud. Evaluate with F1 score, Precision@threshold, Recall, and AUC-ROC. Also check the confusion matrix for the minority class performance.
Explain RAG in 30 seconds. What problem does it solve?
Embed your documents and store in a vector database. When a user asks a question, embed the question, search for semantically similar document chunks, inject those chunks into the LLM prompt alongside the question. The LLM generates an answer grounded in real retrieved data rather than guessing from memory. Solves: hallucination, knowledge cutoff, lack of company-specific knowledge — without retraining.
What is temperature in LLM inference and when would you set it to 0?
Temperature controls randomness by scaling logits before softmax. Temperature=0 always picks the highest probability token (fully deterministic). Set it to 0 for: code generation, structured data extraction, financial calculations, any task requiring reproducibility. Use higher temperatures for brainstorming, creative writing, or when you want diverse outputs.
Random Forest vs XGBoost — what is the fundamental difference?
Random Forest uses bagging: trees trained INDEPENDENTLY on random data subsets, then vote (parallel). XGBoost uses boosting: trees trained SEQUENTIALLY where each tree fixes the errors of the previous one. XGBoost usually achieves higher accuracy but is more sensitive to hyperparameters and can overfit. Random Forest is a safer default; XGBoost is worth tuning when you need maximum accuracy on tabular data.
Explain the transformer attention mechanism. What are Q, K, V?
Q (Query) = what this token is looking for. K (Key) = what information each token offers. V (Value) = what information each token actually provides. Compute attention scores: Q dot K for every token pair. Scale by sqrt(d) to prevent vanishing gradients. Apply softmax to get a probability distribution. Multiply by V to get a weighted sum of values. Result: each token is represented as a context-aware mixture of all other tokens.
When would you choose fine-tuning vs RAG vs just prompting?
Prompting: quick, no training, good for simple tasks where the model already has the knowledge. RAG: need company-specific or dynamic data, no training needed, bottleneck is retrieval quality. Fine-tuning: need a specific style, format, or domain vocabulary baked in; worth the compute when behavior needs to be consistent across thousands of calls. In practice: try prompting first, then RAG, then fine-tuning.
What is backpropagation and how does it work?
After the forward pass (prediction), calculate the loss (how wrong). Use the chain rule of calculus to flow the error BACKWARDS through the network: compute how much each weight contributed to the final error (gradient). Then update each weight proportionally: weight -= learning_rate * gradient. This is gradient descent applied via the chain rule across many layers in sequence.
What is data drift and how would you detect it in production?
Data drift is when the distribution of incoming data shifts from the training data distribution. Detection: monitor feature statistics (mean, std, percentiles) and compare to training baseline. Use KS test or PSI (Population Stability Index) for statistical tests. Also monitor prediction distribution — a sudden shift in output distribution often signals input drift. Set up alerts and retrain triggers.
Why does GDPR matter for ML systems? Name two concrete impacts on a UK ML pipeline.
1. Training data must have a lawful basis for processing — you cannot just scrape personal data to train a model. Data minimisation means only collecting features necessary for the task. 2. Right to erasure ("right to be forgotten") — if a user requests deletion, you may need to retrain the model without their data or implement machine unlearning, since the data may be encoded in model weights. Also: Article 22 requires explainability for automated decisions affecting individuals (loans, hiring, benefits).
What is the difference between interpretability and explainability in ML?
Interpretability: the model structure is inherently human-understandable without additional tools. Decision trees, linear/logistic regression are interpretable — you can trace exactly why a prediction was made. Explainability: the model is a black box but you add post-hoc tools to explain specific predictions. SHAP assigns feature attribution values. LIME builds a local linear approximation. GradCAM highlights relevant image regions. UK finance regulators (FCA) often require explainability for credit scoring.
Explain BM25 vs dense embedding retrieval. When would you use each?
BM25: keyword-based, probabilistic extension of TF-IDF, fast, no GPU, great for exact term matching. Dense retrieval: neural embeddings + cosine similarity, captures semantic meaning, handles paraphrasing and synonyms. BM25 wins when exact keywords matter (product codes, legal terms, error codes). Dense wins when meaning matters more than exact words. Hybrid search (BM25 + dense + RRF reranking) typically beats either alone in production RAG systems.
What is function calling / tool use in LLMs? How does it work mechanically?
You define tools as JSON schemas (name, description, parameters). The LLM receives the tool definitions in its system prompt or context. When it decides a tool is needed, it returns a structured tool_call object with the function name and arguments instead of free text. Your application code parses this, executes the actual function, and returns the result back to the model to continue reasoning. The model never executes code directly — you do the execution and feed results back.
How do you handle class imbalance? Name three techniques.
1. Resampling: SMOTE (oversample minority class by synthesizing new examples between real ones) or random undersampling. 2. Class weights: tell the model to penalize minority class errors more (class_weight='balanced' in sklearn). 3. Focal loss: down-weights easy examples and up-weights hard/rare ones — used in object detection, effective for extreme imbalance. 4. Threshold tuning: default 0.5 threshold may be wrong; optimize using precision-recall curve for your use case. Always evaluate with F1 or AUC-ROC, not accuracy.
What is vLLM and why is it used for LLM serving at scale?
vLLM is a high-throughput LLM inference engine. Key innovation: PagedAttention — manages KV cache memory like an OS manages virtual memory pages, dramatically reducing memory fragmentation. Continuous batching: as soon as one request finishes, a new one joins the batch without waiting — improves GPU utilization from ~20% (naive serving) to 80%+. Used when you need to serve LLMs to many concurrent users without paying for proportionally more GPUs.
